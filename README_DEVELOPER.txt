# Tsukuyomi 開発者向けドキュメント (README_DEVELOPER)

このプロジェクトを今後メンテナンスする開発者（またはAIエージェント）向けに、アーキテクチャ設計、処理のフロー、注意が必要な実装上の制約についてまとめました。

---

## 1. ディレクトリとファイルの役割構成

ソースコードは `src` フォルダ以下に役割ごとに整理されています。

*   `src/dllmain.cpp`
    *   DLLのエントリーポイント。起動時に初期化用スレッド（`MainThread`）を起動し、アンロード時にフックの解除や非同期スキャンスレッドのクリーンアップを行います。
*   `src/console/` (`console.h` / `console.cpp`)
    *   RichEditコントロールを用いた独自コンソールUI（チートメニューとログ表示）を制御します。
    *   メニューのキー操作（矢印キー、Enter、Esc）、設定値の保存/ロード、カラーログ出力を担当します。
*   `src/hotkey/` (`hotkey_monitor.h` / `hotkey_monitor.cpp`)
    *   ゲーム中（非コンソール操作時）のホットキー入力（コンソール展開キー、FreeCameraトグル等）を非同期監視します。
*   `src/hooks/` (`hooks.h` / `hooks.cpp`)
    *   MinHookライブラリを使用したメモリフック（ゲーム関数のインターセプト）の登録・有効化・一括解除を管理します。
*   `src/scans/` (`scans.h` / `scans.cpp`)
    *   libhatライブラリによるメモリシグネチャスキャン処理を一括管理します。スキャンした実アドレスはグローバル変数として公開されます。
*   `src/utils/` (`window_utils.h` / `window_utils.cpp`)
    *   ゲームがアクティブウィンドウであるか（チートコンソール自身や別ウィンドウを除く）を判定する `IsGameFocused()` を提供します。
*   `src/modules/`
    *   機能別のモジュールが配置されています。
    *   `FreeCamera`: カメラ座標制御と移動処理。
    *   `creativeNoClip`: 壁抜け（NoClip）フラグのメモリパッチ。
    *   `AntiDarkness`: 暗闇エフェクトバイパスのためのNOPパッチ。
    *   `SchematicControl`: 回路操作用のOffset Y書き換えフック。

---

## 2. 重要な設計仕様とアーキテクチャ上の制約

### ① 起動時の設定ロードとアドレス特定（初期化順序の制約）
*   **挙動の流れ**:
    1.  DLLロード時、`CreateCustomConsole` 内でコンソールウィンドウが作成されます。
    2.  ウィンドウの `WM_CREATE` 時点で `InitMenu()` が走り、各モジュールの設定ファイル（`.json`）から状態がロードされ、`SetCreativeNoClipEnabled()` などのAPIが即座に呼び出されます。
    3.  **注意**: このロード時点では、バックグラウンドでのシグネチャスキャン（`PerformSignatureScans()`）がまだ完了しておらず、対象のアドレス（`g_addrCreativeNoClip` 等）はすべて `nullptr` です。
*   **実装上の工夫**:
    *   各モジュールの `Set...Enabled(bool)` は、アドレスポインタが `nullptr` であっても、まず状態を表すグローバル変数 `g_enabled = enabled;` を更新した上で、早期 `return` するように設計されています。
    *   その後、スキャンが成功したタイミング（`dllmain.cpp:25`）で `Set...Enabled(Get...Enabled())` を再実行することにより、保持されていた設定値に基づいて実際のメモリ領域にフックやパッチが安全に適用されます。
    *   **警告**: 新しいモジュールを追加したり、有効化関数を書き換える際は、アドレスが `nullptr` でも状態変数 (`g_enabled`) が確実に更新される構造を維持してください。

### ② 同期ブロックによる画面フリーズの回避（スレッド間メッセージ通信）
*   **問題**:
    *   ゲームのメインスレッド（描画やフック処理中）から、UIコントロールを操作する `AddLog()` を同期的に呼び出すと、RichEditコントロールへの `SendMessage` が処理を完了するまでゲーム側が同期ブロックされ、ゲーム画面が一瞬カクつく（フリーズする）致命的な問題が発生しました。
*   **対策**:
    *   `AddLog` 内部で `GetWindowThreadProcessId` を使用し、呼び出し元のスレッドIDがコンソールUIスレッド（`g_hMainWnd` の所有スレッド）と異なるかチェックします。
    *   UIスレッド以外からの呼び出しの場合、ログテキストをヒープ領域にコピーし、`PostMessageW` を用いて非同期カスタムメッセージ `WM_USER_ADD_LOG` をコンソールウィンドウにポストし、即座にリターンします。
    *   UIスレッド側の `WndProc` で `WM_USER_ADD_LOG` を受信した際に、同期的に `AddLog` を実行してRichEditにログを挿入し、最後にヒープバッファを安全に `delete` します。
    *   **警告**: ゲームスレッドからコンソールUIへアクセスする処理を追加する場合は、ゲーム画面をフリーズさせないために、必ずこの非同期 Post 構造を使用してください。

### ③ アクティブ判定による誤入力の防止
*   *   各種キー判定（FreeCameraの移動キーや、SchematicControlの矢印キーによるOffset Y操作など）を行う際は、必ず `IsGameFocused()` を使ってゲーム画面がアクティブ（かつチートコンソール自身が開かれていない）であることを条件に組み込んでいます。
    *   これにより、コンソールに入力している際や、Alt+Tabで別の作業をしている間に、意図しないチートキー入力を受け付けることを防止しています。

### ④ SchematicControl の非同期スキャンとアンロード時処理
*   *   `SchematicControl` のシグネチャスキャンは、機能が有効（ON）な間のみ、別スレッド `g_scanThread` を立てて非同期でループ実行し続けます。これにより、起動時にスキャン待ちによるハングが発生するのを防止しています。
    *   DLLがアンロードされる際、バックグラウンドスレッドがゾンビ化してクラッシュするのを防ぐため、`dllmain.cpp` の終了時に `CleanupSchematicControl()` を明示的に呼び出し、スレッドを安全に `join` させてフックを全解除しています。
